# Gotchas

Things that have bitten before and will bite again.

## Angel One SmartAPI

- **`smartapi-python` has incomplete dependency declarations (2026-06-06)**: `pip install smartapi-python` does NOT pull in `logzero` and `websocket-client`, but the package imports both at load time → `ModuleNotFoundError` on first import. Fix: `pip install smartapi-python pyotp python-dotenv logzero websocket-client`. **Add all five to `requirements.txt`** or the Cloud Run deploy will fail the same way. See [[Trading System]].
- **Segment codes are non-obvious (2026-06-06)**: `getProfile()` returns segments as `mcx_fo` (MCX commodity), `cde_fo` (currency derivatives / EUR/USD), `nse_fo`, `nse_cm`, etc. — NOT `"MCX"` / `"CDS"`. Match the `_fo`/`_cm` codes when checking segment activation.
- **TOTP needs the base32 SECRET, not the 6-digit code (2026-06-06)**: store the secret string from TOTP setup in `.env` as `ANGEL_ONE_TOTP_SECRET`; generate the rolling code at runtime with `pyotp.TOTP(secret).now()`. SmartAPI `generateSession(client, pin, totp)` — 2nd arg is the **MPIN**, 3rd is the generated code.

## Telegram signal bot

- **Callback-data float precision collapses FX prices (2026-06-12)**: the scanner's inline keyboard encoded entry/SL/TP into `callback_data` at `:.2f`. Fine for gold (4678.10) but for EUR/USD (1.15356 → "1.15") the entry, SL, and TP all round to nearly the same 2-dp value — the downstream Validate/Execute handler then trades against a corrupted price. Fix: format by instrument — `decimals = 2 if cfg["unit"] >= 1.0 else 5`, threaded through `send_signal_keyboard` and `_record_paper_trade`. Telegram `callback_data` hard limit is **64 bytes** — the 5-dp FX payload measures 51 bytes, safe, but check this whenever adding fields. Any new low-priced instrument (most FX, many alts) inherits this trap. See [[Trading System]].

## Angel One execution stub

- **`bot/execution.py` stub comments name the wrong env vars (2026-06-12)**: the commented-out `SmartConnect` block references `ANGEL_ONE_PASSWORD` and `ANGEL_ONE_TOTP`, but the actual `.env` (and `test_angel_login.py`) use `ANGEL_ONE_PIN` and `ANGEL_ONE_TOTP_SECRET` (base32 secret → `pyotp.TOTP(secret).now()` at runtime). Wire the real names when un-stubbing or login silently fails with a confusing error.
- **`placeOrder` ignores `stoploss`/`squareoff` on `variety="NORMAL"` (2026-06-12)**: the stub's order params include `stoploss` and `squareoff`, but those only take effect on bracket/robo varieties — on a NORMAL market order they're silently dropped, leaving the position with **no protective stop**. When wiring live orders, place the SL as a separate order (or use the correct variety). This is a money-losing trap, not a cosmetic one.

## Claude Code CLI

- **PowerShell mangles the `--` subprocess separator (2026-06-18)**: `claude mcp add gmail -- npx -y @gongrzhe/server-gmail-autoauth-mcp` in PowerShell threw `unknown option '-y'` — PowerShell isn't passing `--` through as a literal separator to the CLI's arg parser. Tried `--%` stop-parsing token and the `&` call operator, both still failed the same way. **Fix: run `claude mcp add ... -- <command>` through the Bash tool** (POSIX sh), which preserves `--` correctly. Applies to any `claude mcp add` call with subprocess args/flags after `--`.

## CoinDCX REST API

- **A *generic* `400 Invalid request` on `/orders/create` is NOT necessarily a payload bug — and NOT necessarily a key-permission bug (2026-06-13)**: hit this hard. Reads (balances, market data) all work; every order write returns the same generic 400 regardless of payload. The *uniformity* tempted a "read-only key" diagnosis — **wrong** (the dashboard showed Trading permission enabled + IP whitelisted). Systematically ruled out: auth, trade permission, IP, market identifier (`"BTCUSDT"` is correct — pair form `"B-BTC_USDT"` gives a *different* `422`, proving the name is accepted), clock skew, string-vs-float numbers, `ecode` field, price-band. Our client was confirmed **byte-identical to the working `svamja/coindcx-python` reference lib**. Leading cause: the **`B`-ecode (Binance-mirrored) USDT markets may not be REST-orderable** for the account despite `status:active`, or an **account-level API-trading activation** is needed. **Lesson: "uniform failure ⇒ X" is a weak inference** — uniformity fits payload bugs, permission bugs, AND account-side blocks equally. When our code matches a known-working reference, stop guessing and go to exchange support. See [[CoinDCX Execution Layer]].
- **CoinDCX Help docs list `key` and `secret` as required body parameters for `/orders/create` — misleading (2026-06-17)**: tested adding `key` to the request body → still 400. Cross-verified against the `svamja/coindcx-python` reference library: auth is **headers only** (`X-AUTH-APIKEY` + `X-AUTH-SIGNATURE`). The docs say "specify the following details in the body of your code" meaning "you need API credentials in your config" not "put them literally in the JSON payload." Adding `key` to the body changes the HMAC-signed payload without fixing anything. See [[CoinDCX Execution Layer]].
- **Small quantities serialize to scientific notation and break orders — FIXED (2026-06-13)**: `json.dumps(0.00005)` → `"5e-05"`, which CoinDCX rejects. Any quantity **< 0.0001** (including the `1e-05` market minimum and any min-notional-sized BTC order) hits it; offline tests with `0.0012`-size quantities won't catch it. Fix in `coindcx_client.py`: `_plain_decimal()` + `_signed_post(number_fields=...)` emit the fields as plain non-scientific **JSON numbers** (raw, unquoted — matches the reference lib) via placeholder injection. 17 tests green. See [[CoinDCX Execution Layer]].

- **`take_profit_limit` does not exist on CoinDCX — the correct type is `take_profit` (2026-07-01)** — `[VERIFIED]`, money-touching. Built Phase 2 bracket with `take_profit_limit` throughout; `422: "order_type must be one of [limit_order, market_order, stop_limit, take_profit, stop_limit_market, take_profit_market]"` on first live test. The full valid set: `limit_order`, `market_order`, `stop_limit` (trigger → limit sell below trigger), `take_profit` (trigger → limit sell below trigger), `stop_limit_market` (trigger → market sell), `take_profit_market` (trigger → market sell). Always live-test a new CoinDCX order type with a tiny cancel-immediately order before wiring into production. Catch: both `stop_limit` and `take_profit` use `stop_price` (trigger) + `price_per_unit` (limit), with constraint `stop_price > price_per_unit` for both. See [[CoinDCX Execution Layer]].
- **yfinance daily bar replay is blind to intraday CoinDCX stop fills (2026-07-01)** — `[VERIFIED]`, money-touching. The monitor runs once per day at 06:30 IST. If a stop_limit SL triggers at 2am, yfinance has not yet closed a daily bar that reflects it — monitor sees nothing, does nothing, TP order stays live against a closed position. If BTC then rallies and TP fills, that's a naked short sell. **Fix: always poll `get_active_orders()` at monitor start for any trade with sl/tp order IDs.** An order absent from the active set = it filled (or was cancelled). yfinance stays only as the max_hold backstop. This failure mode is the base case for any intraday move, not an edge case. See [[CoinDCX Execution Layer]].

## Perplexity / API cost tracking

- **A flat per-call cost estimate hid a ~15x real spend undercount (2026-06-18)**: `audit.py`'s `COST_TABLE` assumed every Perplexity call cost a flat `$0.005`, regardless of model or response length. Real billing dashboard showed `$1.05/30 days` against a ledger estimate of `~$0.05-0.07` for the same window — output tokens (not request count) are the actual cost driver, and `sonar-pro` output is 15x `sonar`'s rate ($15/M vs $1/M). Compounding it: `research.py` and `research_deep.py` never called `vault.append_to_ledger()` at all — only `finance.py` self-logged — so most real calls were invisible to the audit regardless of the pricing bug. **Lesson: a cost ledger that estimates instead of measuring will silently drift from reality; periodically reconcile against the provider's real billing dashboard.** Fixed — see [[Key Decisions#Research & Intelligence Infrastructure]] and [[api-decision-framework#Cost Optimization Strategy (applies to ANY paid API — Perplexity, Grok, Gemini, future ones)]].

## Research scripts (Windows)

- **`UnicodeEncodeError` on ₹ / unicode (2026-06-05)**: `/research` (Perplexity) output containing `₹` crashes on Windows cp1252. Fix: set `$env:PYTHONUTF8 = "1"` before running. Same fix applies to running any trading script that prints `₹`/`—` — pass `PYTHONIOENCODING=utf-8`.
- **Now fixed at source in the obsidian-second-brain toolkit (2026-06-14)**: the per-run `PYTHONUTF8` workaround is no longer needed for the research scripts. `lib/config.py` reconfigures `stdout`/`stderr` to UTF-8 on import (covers every paid script that imports it), and `lib/vault.py` `write_note`/`append_to_log`/`append_to_daily` now write files with `encoding="utf-8"`. **Two distinct cp1252 traps, both bit this session**: (1) `print()` of unicode to the console (the old known one), and (2) `Path.write_text(s)` *defaulting* to cp1252 when saving notes — the second silently lost a finished Perplexity dossier because the crash fired *before* the save. **Lesson: on Windows, always pass `encoding="utf-8"` to every file write and reconfigure stdio; never rely on the platform default.** A subprocess that emits UTF-8 also needs the *caller* to read with `encoding="utf-8"` (or route emoji/progress to stderr so machine-mode stdout stays ASCII) — see the `vault_health.py --json` fix. Building the `/finance` tool ([[api-decision-framework]]) surfaced all of these.

## GitHub Actions CI

- **Actions auth runs on the `GH_PAT` repo secret, NOT your local git credentials (2026-06-17)** — `[VERIFIED]`. When the leaked OAuth token was revoked on 2026-06-16, re-authenticating *local* git fixed nothing in CI: all four life-os workflows that checkout with `token: ${{ secrets.GH_PAT }}` (`crypto_monitor`, `crypto_scanner`, `monitor`/Paper Trade Monitor, `scanner`/Trading Scanner) began failing at the **checkout** step (~40s, before the script even runs). - **The fix is to drop the PAT entirely and use the built-in `GITHUB_TOKEN` (2026-06-17)** — `[VERIFIED]`, shipped (life-os `ea11bf7`). If a push still 403s, flip repo **Settings → Actions → General → Workflow permissions → "Read and write."**
- **"Re-run jobs" replays the OLD commit — it does NOT pick up a pushed fix (2026-06-17)** — `[VERIFIED]`, cost a confused round-trip. After pushing the `GITHUB_TOKEN` fix, clicking **Re-run jobs** on the failed run re-executed the *previous* commit (still on `GH_PAT`) and failed identically, making the fix look broken. **Always use the "Run workflow" dispatch button (on branch `main`) to test a pushed change** — a fresh dispatch runs the latest code; a re-run is pinned to the run's original SHA. ## Vault tooling

- **Mixed CRLF/LF line endings silently break the Edit tool's exact-match (2026-06-16)** — `[VERIFIED]`. The Edit tool's `old_string` is matched against LF-normalized text, but several vault `.md` files were CRLF (or mixed CRLF+LF after a Bash `>>` append). Multi-line/multi-word anchors then fail to match byte-for-byte even though the Read output looks identical — three Edits bounced off `Patterns.md` this way before the cause was found. **Diagnosis:** `file <note>.md` reports "with CRLF, LF line terminators"; `tr -cd '\r' < f | wc -c` counts CR bytes (should be 0). **Fixed permanently:** added root `.gitattributes` with `*.md text eol=lf`, and normalized the offenders (`tr -d '\r'`). Because `core.autocrlf=true` already stores blobs as LF, normalizing the working tree produces **zero content diff**. **Lesson: if an Edit keeps failing to match a string the Read clearly shows, suspect line endings before re-reading — and don't append to `.md` with Bash `>>` (it injects LF into a CRLF file); use the Write/Edit tools.** See [[Patterns#Automation routing — who drives, what's the token pool|Patterns]].

## PF Withdrawal & Transfer

- **Transfer PF, don't withdraw, when switching jobs** — each employer opens a PF account under your UAN. When you leave, transfer the old balance to the new employer's PF account (EPFO portal → One Member One EPF Account → Transfer). The balance keeps earning ~8.25% interest and the 5-year clock for tax-free withdrawal continues. Withdrawal mid-career is taxed if total corpus > ₹50,000 and employment < 5 years.
- **Activate UAN on Day 1** — link Aadhaar + bank account immediately via the EPFO member portal (epfindia.gov.in). Without this, online withdrawal/transfer is blocked.
- **Withdrawal process (if you do withdraw)** — wait 2 months after last working day → EPFO portal → claim online → money in ~15-20 days. No physical paperwork needed.
- **CTC inflation** — PF Employer (₹1,800/month) + Gratuity accrual (₹1,803/month) are both included in CTC but aren't liquid. PF employer is real money (yours on exit); gratuity requires 5 years to vest, so for short stints it's phantom CTC.
